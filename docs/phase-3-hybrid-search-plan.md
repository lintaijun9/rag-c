# RAG Phase 3 - PgVector Hybrid Search 集成及改进计划

基于前面的基础搭建，底层服务现已全部通过 Drizzle ORM 以及强类型安全接管了 postgres 通信。在此为您提出接下来的 **第三阶段（最关键的 AI 语义强化阶段）** 改进执行方案。

## 为什么需要在这个阶段引入混合检索？
当前查询完全基于 SQL 的 `ILIKE` 通配符匹配（Keyword-based Search）。当数据库内容高达上万条长文本数据时：
1. **关键词硬匹配的遗漏率很高：** 比如用户搜“医疗建议”，系统很难通过文字直接找回只包含“如何抗癌与自我调理”的内容，因为字面上没有“医疗”。
2. **结合 Gemini 增强 RAG 的威力：** 尽管后台可以使用 Gemini 官方自带的 File Search，但如果在本地数据库就利用向量检索，把精准的片段找出来再发给 Gemini 归纳计算（Grounding），既能省去大量的官方 API 请求时长（Latency），也会有极高精度的表现反馈。

---

## 具体架构演进与集成方案 (10条技术实施路径)

### 1️⃣ 开启 Postgres 内部引擎的向量支持
我们将在数据库实例底层安装 Vector 插件：通过预配脚本执行 `CREATE EXTENSION IF NOT EXISTS vector;` 以保证 DB 完全觉醒多维数组的算力。目前我们在 `src/db/schema.ts` 已经预置了对应的 768 维 `embedding` 隐式字段和 `hnsw` 超快查询索引。

### 2️⃣ 搭建嵌入任务队列 (Embedding Pipeline) 
数据库中现有的 8000 多条历史数据尚未进行嵌入。需要开发一个离线或后台 Node.js 修复脚本（Backfill Script），通过 Google `@google/genai` 的 `embedContent` 批量对现存 `content` 和 `summary` 生成向量并刷回数据库，每次处理 100 条。避免遇到调用上限。

### 3️⃣ 设计混合搜索算法服务 (Hybrid Search Algorithm) 
在 `src/services/rag.service.ts` 中整合**动态权重逻辑**:
- 使用 Drizzle 构建**关键字相似度 (BM25 近似)** + **语义余弦相似度 (Cosine Similarity)** 双重打分机制 $(\alpha \times \text{Keyword}) + ((1 - \alpha) \times \text{Semantic})$，以保障能像 Google 一样搜得既准确又聪明。

### 4️⃣ 重构查询分析引擎 API (`api/rag/query`)
拦截旧版的单向文本传递。用户发送的 Question 将被首先转化为一段局部 embedding 向量数组，作为比对种子发送到 postgres 并直接锁定最接近上下文内容的 5 条结果。

### 5️⃣ Reranking 重排序机制 (引入更成熟的评估精度)
前端搜出的混合结果会通过算法在 Node 层面上对内容进行交叉计算。将最终得出的最强前 k 条传递给 Gemini 执行归纳。以保障生成内容不出幻觉。

### 6️⃣ 降低 Gemini Server 端 Token 使用率成本
我们将过滤掉无用上下文。在传给 Gemini 补全时，我们现在不需要全文堆叠传递，而是只把 pgvector 返回的 `start_time` 切片内容进行精装合并，节省超过 60% 的隐形 Token 计费。

### 7️⃣ RAG 数据同步监听链路  
修改现有的 webhook / 上传脚本（比如从 `/scripts` 脚本上传时），增加一步自动化机制。当有新的会议记录入库或导入时，能够触发异步钩子，后台无痛为其绑定并生成嵌入内容存储至 `embedding` 字段。

### 8️⃣ 前端混合搜索的感知交互
为 `RagAiTab.tsx` 和主页面的 `Search` 打通开关，如增加“语义匹配模式”、“严格模式”供用户选择。也可以对呈现的最佳匹配项目加上诸如 `“相关度 98%”` 的可信度标签。

### 9️⃣ 前端的骨架屏和容错策略
由于查询向量可能比传统查询慢20-50ms左右并在极小概率下断链，我们将在 SWR 中加入悬浮骨架屏与失败重试策略。如果向量化因为 API 错误挂掉，系统可以自动回退（Fallback）为普通的 ILIKE 搜索来确保服务的强健性。

### 🔟 全面日志增强与断点告警分析  
将 `logger` 加入全部的语义请求中（比如记录 `Question: "如何看病" 召回 ID [10, 442, 88]`），结合以前端报错最多的 `api/rag/stats` 返回的状态构建链路监控，防止产生因为网络丢包导致不明 500 报错时毫无头绪地排查。
